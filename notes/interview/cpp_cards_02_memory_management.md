# C++ 卡片 02：内存管理（50）

1. **栈与堆**：栈内存由编译器自动管理，分配释放快但容量有限。堆内存手动/间接管理，更灵活但易泄漏。
2. **自动存储期**：局部对象离开作用域会自动析构。RAII 的安全性主要依赖这条规则。
3. **动态存储期**：`new` 出来的对象生命周期独立于作用域。必须确保最终有唯一清晰的释放路径。
4. **`new/delete` 与 `malloc/free`**：前者负责构造/析构，后者只做原始内存分配。两组 API 不能混用。
5. **`new[]/delete[]` 配对**：数组分配必须用数组释放。配错会导致析构次数错误或未定义行为。
6. **内存泄漏**：分配后丢失释放路径即泄漏。长期运行服务会被慢慢“吃空”内存。
7. **double free**：同一块内存释放两次会破坏分配器状态。典型原因是浅拷贝+手动释放。
8. **use-after-free**：释放后继续访问指针指向区域。可能“偶尔正常”，但本质是高危未定义行为。
9. **悬垂指针**：指针仍有地址但目标对象已不存在。应在释放后置空并减少裸指针拥有权。
10. **野指针**：未初始化指针指向随机地址。初始化为 `nullptr` 是最低成本防线。
11. **所有权模型**：先定义“谁拥有、谁释放”。设计清晰时，内存 bug 会显著减少。
12. **唯一拥有权**：同一资源只允许一个拥有者，语义最简单。`std::unique_ptr` 是默认优先选择。
13. **共享拥有权**：多个对象共同拥有资源，最后一个释放者清理资源。适合生命周期跨模块共享场景。
14. **观察者语义**：只引用不拥有，不能决定释放时机。裸指针/引用常用于表达观察者角色。
15. **`unique_ptr` 基本语义**：不可拷贝、可移动，确保唯一所有权。接口层面能直接阻止误复制。
16. **`unique_ptr` 与工厂**：优先用 `std::make_unique` 创建，异常安全更好。避免先 `new` 再包裹的窗口期风险。
17. **`unique_ptr` 自定义删除器**：可管理文件句柄、socket 等非 `delete` 资源。让 RAII 扩展到系统资源。
18. **`shared_ptr` 语义**：通过引用计数共享对象生命周期。复制 `shared_ptr` 会增加计数。
19. **控制块（control block）**：保存引用计数、删除器等元信息。理解它有助于解释性能开销。
20. **`make_shared` 优势**：对象与控制块通常一次分配，减少分配次数与碎片。一般优先于 `shared_ptr(new T)`.
21. **`shared_ptr(new T)` 风险**：对象和控制块分开分配，异常路径更复杂。通常不如 `make_shared` 简洁安全。
22. **循环引用问题**：`shared_ptr` 相互持有会让计数永不归零。最终表现为“逻辑不再使用但不释放”。
23. **`weak_ptr` 作用**：打破共享环，表达“可观测但不拥有”。`lock()` 成功才说明对象仍存活。
24. **`weak_ptr::expired`**：可快速判断资源是否已释放。并发场景仍应用 `lock()` 做原子检查+获取。
25. **`enable_shared_from_this`**：对象内部安全获取自身 `shared_ptr`。前提是对象本来就由 `shared_ptr` 管理。
26. **错误的 `shared_from_this`**：若对象不在 `shared_ptr` 控制下调用会抛异常。构造阶段调用也常是坑点。
27. **引用计数线程语义**：计数增减通常是线程安全的。对象本身的数据读写是否安全是另一回事。
28. **`shared_ptr` 性能成本**：原子计数和控制块带来开销。高频路径不该无脑使用共享拥有权。
29. **裸指针的定位**：可用于非拥有访问、参数传递、与 C API 交互。不要把它当默认资源拥有者。
30. **引用的定位**：表达“必须非空”的别名语义。不能 reseat，适合稳定生命周期对象。
31. **`nullptr` 语义**：类型安全的空指针字面量。优于 `NULL` 和 `0` 的重载解析行为。
32. **内存对齐与分配**：对象地址要满足对齐要求，否则性能下降甚至硬件异常。自定义分配器时必须考虑。
33. **placement new 场景**：在预分配缓冲区上构造对象。常用于内存池与对象池。
34. **显式析构调用**：placement new 构造的对象需要手动调用析构。否则资源不会被正确释放。
35. **内存池价值**：减少频繁系统分配带来的抖动。适合大量小对象、低延迟场景。
36. **碎片化问题**：长期随机分配释放会让可用内存被切碎。池化和对象复用能降低碎片。
37. **越界写危害**：会破坏相邻对象或分配器元数据。排查时常表现为“晚发生、难复现”。
38. **ASan 工具价值**：可快速定位越界、UAF、double free。面试提到 sanitizer 能体现工程意识。
39. **Valgrind 定位泄漏**：在 Linux 下可追踪泄漏来源。代价是运行慢，但排错信息清晰。
40. **异常安全释放**：构造中间步骤失败时也要确保资源回收。RAII 容器比手动回滚可靠。
41. **资源句柄封装**：文件、锁、socket 都应封装成小类。把“获取-释放”固定在构造/析构里。
42. **拷贝与拥有权**：拥有资源的类若允许拷贝，必须定义清晰复制策略。否则很容易触发双重释放。
43. **移动与拥有权**：移动后源对象应置于可安全析构状态。常见做法是把句柄置空。
44. **观察者失效**：被观察对象释放后，观察者引用应立即失效。`weak_ptr` 比裸指针更易表达这点。
45. **容器失效规则**：`vector` 扩容会使指针/引用/迭代器失效。失效后继续使用属于未定义行为。
46. **跨模块分配释放**：谁分配谁释放是基本规则。跨 DLL/运行库边界尤其要避免混配分配器。
47. **C API 封装策略**：把 `malloc/free` 资源封进智能指针并指定删除器。减少桥接代码出错概率。
48. **内存模型与所有权**：先解决“生命周期正确”，再谈并发正确。生命周期错误会放大并发 bug。
49. **默认实践**：优先值语义与容器，必要时 `unique_ptr`，最后才 `shared_ptr`。这是多数项目更稳的选择。
50. **面试回答重点**：不要只背 API，先讲所有权图，再讲失败路径和工具验证方式。
