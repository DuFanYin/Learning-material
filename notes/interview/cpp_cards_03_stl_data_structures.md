# C++ 卡片 03：STL 与数据结构（50）

1. **容器选择原则**：先按访问模式选容器，再看复杂度。错误容器选择往往比微优化影响更大。
2. **`vector` 本质**：连续内存动态数组，随机访问快。缓存友好通常让它在实践里很强。
3. **`vector` 扩容**：容量不足会重新分配并搬迁元素。旧迭代器和指针通常全部失效。
4. **`vector` 预留容量**：`reserve` 可减少重复扩容。对已知规模数据能明显降抖动。
5. **`vector` 与 `list`**：理论上插入复杂度不同，但很多场景 `vector` 仍更快。原因是局部性和更少指针跳转。
6. **`deque` 结构**：分段连续，不是单块连续。头尾插入高效且稳定。
7. **`deque` 使用场景**：频繁头尾操作时优于 `vector`。典型如滑动窗口队列。
8. **`list` 结构**：双向链表，节点离散分配。随机访问慢且 cache 命中差。
9. **`list` 价值**：迭代器在节点不删时稳定，节点级 splice 成本低。适合特定结构性操作。
10. **`array` 语义**：固定大小的栈上数组封装。零额外开销且能与 STL 算法配合。
11. **`string` 本质**：动态字符容器，接口接近 `vector<char>`。注意编码和字节长度不是同一概念。
12. **SSO（小字符串优化）**：短字符串存放在对象内部，避免堆分配。能显著降低小文本场景开销。
13. **`map` 结构**：通常红黑树，实现有序键值存储。查找/插入/删除一般 O(log n)。
14. **`unordered_map` 结构**：哈希桶+链/开放寻址，平均 O(1)。负载和哈希质量决定表现。
15. **哈希退化**：冲突严重时 `unordered_map` 可能接近 O(n)。面试要提及最坏复杂度风险。
16. **有序 vs 无序映射**：需要排序遍历选 `map`，追求平均查询性能选 `unordered_map`。别只看大 O，要看业务需求。
17. **`set` 与 `unordered_set`**：分别对应有序去重与哈希去重。语义与 map 类似但只存 key。
18. **`priority_queue`**：默认大顶堆，取极值高效。常用于 TopK、调度和图算法。
19. **`stack`/`queue` 适配器**：它们是容器适配器，不是独立底层结构。默认底层分别常用 `deque`。
20. **迭代器分类**：输入/前向/双向/随机访问迭代器能力不同。算法对迭代器能力有前置要求。
21. **迭代器失效**：不同容器规则不同，必须单独记忆高频容器。失效后继续使用是未定义行为。
22. **`erase` 返回值**：很多容器 `erase` 会返回下一个有效迭代器。遍历删除时应使用返回值推进。
23. **范围 for 与引用**：`for (auto x : v)` 会拷贝元素。修改元素应使用 `auto&`。
24. **`emplace` vs `push`**：`emplace` 原地构造可减少临时对象。对复杂对象更有价值。
25. **`insert` 位置成本**：`vector` 中间插入需搬迁后续元素。大规模中间插入应谨慎。
26. **`shrink_to_fit`**：请求收缩容量但不保证一定发生。不能依赖其严格语义。
27. **算法优先思维**：优先复用 `<algorithm>`，代码更短且更少 bug。比如 `sort/find_if/partition`。
28. **`sort` 要求**：随机访问迭代器，通常是 introsort。比较器必须满足严格弱序。
29. **比较器陷阱**：比较器不自反/不传递会导致排序行为异常。面试常问“为什么偶发错序”。
30. **`stable_sort`**：保持相等元素原相对顺序。多关键字排序时很有用。
31. **二分家族**：`lower_bound/upper_bound/equal_range` 建立在有序区间上。无序容器调用语义错误。
32. **`nth_element`**：可线性期望时间找第 k 小。TopK 常比全排序更省。
33. **`partial_sort`**：只关心前 k 个有序时更高效。比全量排序工作量小。
34. **`accumulate` 与类型**：初值类型会影响结果类型。整型累加到 `double` 要显式给 `0.0`。
35. **`remove` 惯用法**：`remove` 只重排不删容器容量。要配合 `erase` 才真正删除。
36. **`transform`**：表达映射关系比手写循环更清晰。可就地转换也可输出到新容器。
37. **`copy_if`**：按条件筛选元素到目标容器。比手写 if-push 更语义化。
38. **`all_of/any_of/none_of`**：快速表达谓词判断。代码可读性优于布尔循环。
39. **`span`（C++20）**：非拥有视图，统一数组/向量接口。避免裸指针+长度参数分离。
40. **分配器意识**：容器默认分配器满足通用场景。极致性能场景可自定义池化分配器。
41. **异常安全与容器**：操作失败时容器应保持有效状态。标准库通常提供清晰异常保证。
42. **移动语义与容器**：元素支持高效移动时，扩容/重排成本可下降。自定义类型最好提供 `noexcept` move。
43. **节点容器与稳定性**：`list/map` 节点地址通常稳定。适合长期持有迭代器或引用的场景。
44. **缓存友好性**：连续存储通常更容易预取和批量处理。CPU 视角常改变“理论最优”结论。
45. **复杂度不是全部**：常数项、分支预测、内存分配同样关键。工程选型要看真实数据分布。
46. **`unordered_map` rehash**：rehash 会重建桶并使迭代器失效。预估容量可减少 rehash 次数。
47. **键类型设计**：键应稳定、可哈希/可比较且语义明确。避免用可变字段做键。
48. **值语义配合容器**：对象可移动可复制时容器更易管理。资源语义混乱会放大容器使用风险。
49. **面试描述模板**：先讲底层结构，再讲复杂度，再讲场景 trade-off。最后补一个常见坑点。
50. **实战检查清单**：是否会扩容、是否迭代器失效、是否需要有序、是否关心内存局部性。
