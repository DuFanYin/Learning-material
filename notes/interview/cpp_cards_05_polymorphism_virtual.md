# C++ 卡片 05：多态与虚函数（50）

1. **多态分类**：C++ 有静态多态（模板）和动态多态（virtual）。两者解决“同接口不同实现”，但时机不同。
2. **动态多态本质**：通过基类指针/引用在运行时分发到派生实现。核心是虚函数机制。
3. **虚函数表概念**：多数实现用 vtable + vptr 完成分发。每次虚调用通常多一次间接跳转。
4. **虚调用成本**：主要是间接调用和潜在的内联受限。多数业务场景可接受，热点路径需评估。
5. **`override`**：显式声明“我要重写基类虚函数”。可在编译期抓住签名不匹配错误。
6. **`final`**：禁止继续重写或继承。用于稳定层次结构和限制误扩展。
7. **虚析构函数必要性**：基类通过指针删除派生对象时，析构必须是 virtual。否则会资源泄漏或未定义行为。
8. **纯虚函数**：`=0` 表示接口契约，基类变抽象类。常用于定义能力边界。
9. **抽象类定位**：强调“做什么”而不是“怎么做”。有利于解耦调用侧与实现侧。
10. **对象切片**：按值传递基类会丢失派生部分。多态接口应优先用引用或指针。
11. **向上转型**：派生到基类通常安全且隐式。是多态调用成立的前提。
12. **向下转型风险**：基类到派生需确认真实动态类型。错误转型会导致未定义行为。
13. **`dynamic_cast`**：运行时安全检查的向下转型工具。失败时指针返回空，引用抛异常。
14. **`static_cast` 向下转型**：无运行时检查，快但危险。只在类型关系绝对可证时使用。
15. **RTTI 概念**：运行时类型信息支持 `dynamic_cast/typeid`。有成本但可提升安全性。
16. **`typeid` 用途**：获取对象动态/静态类型信息（取决于是否多态基类）。一般不应替代多态设计本身。
17. **多重继承**：可表达组合能力，但对象布局更复杂。菱形继承与二义性是高频坑点。
18. **虚继承**：用于解决菱形继承重复基类子对象问题。代价是实现复杂与潜在性能开销。
19. **接口继承 vs 实现继承**：优先把继承用于抽象接口，而非复用实现细节。可降低脆弱基类问题。
20. **组合优于继承**：当“has-a”关系更自然时用组合。继承应保留给明确“is-a”关系。
21. **NVI 模式**：公开非虚接口，内部调用私有虚钩子。能统一前后置逻辑和不变量检查。
22. **模板方法模式**：基类定义流程骨架，派生覆盖步骤。是经典动态多态应用。
23. **策略模式**：把可变算法封到独立策略对象。组合策略比层层继承更灵活。
24. **接口稳定性**：公共虚接口一旦发布，变更成本高。需要前期谨慎设计。
25. **ABI 兼容问题**：跨模块/插件场景中 vtable 布局兼容性很关键。编译器和选项差异会放大风险。
26. **析构顺序**：删除派生对象时先派生析构再基类析构。理解顺序有助于排查资源释放问题。
27. **构造期虚调用陷阱**：构造/析构过程中调用虚函数不会分发到更“外层”派生。语义常与直觉不一致。
28. **默认参数与虚函数**：默认参数静态绑定，虚调用动态绑定。两者混用容易出“参数不一致”困惑。
29. **协变返回类型**：派生重写时可返回更具体的指针/引用类型。能提升接口表达力。
30. **`noexcept` 与重写**：派生重写的异常规范不能更宽。否则会违反替换原则。
31. **访问控制与重写**：`private virtual` 仍可被派生重写。访问权限影响调用点，不影响是否虚分发。
32. **隐藏与重写区别**：同名不同签名可能是隐藏而非重写。`override` 能有效避免此类错误。
33. **`using Base::foo`**：可把基类重载集合引入派生作用域。避免无意隐藏掉其他重载。
34. **纯虚析构也需定义**：即使析构是纯虚函数，通常仍要提供函数体。因为销毁链条会调用到它。
35. **接口最小化**：虚函数数量越多，演进越难。只暴露真正稳定的扩展点。
36. **多态与测试替身**：抽象接口让 mock/stub 更容易。可提升单测隔离能力。
37. **静态多态补位**：当类型在编译期已知，可用模板替代虚函数。在性能敏感模块常见。
38. **类型擦除**：在运行时统一不同类型行为，不依赖继承层次。`std::function` 是典型例子。
39. **虚函数去虚化**：编译器在可证明动态类型时可能去虚。并非所有虚调用都一定慢。
40. **对象内存开销**：多态对象通常包含 vptr。在大量小对象场景要关注额外字节影响。
41. **跨线程多态对象**：多态不等于线程安全。对象状态并发访问仍需同步机制。
42. **生命周期边界**：接口返回裸指针时要写清所有权。更推荐智能指针或引用表达约束。
43. **错误处理策略**：虚接口应统一错误语义（异常/错误码）。避免派生各自为政导致调用侧复杂。
44. **LSP（里氏替换）**：派生类必须可替换基类而不破坏语义。这是继承设计最核心约束。
45. **脆弱基类问题**：基类实现细节改动可能破坏派生行为。减少可覆盖点可缓解风险。
46. **接口文档化**：虚函数要注明前后置条件和线程语义。否则扩展方容易踩坑。
47. **面试常见题**：为何析构要 virtual、虚表如何工作、何时不用继承。回答要带权衡而非口号。
48. **设计优先级**：先清楚变化点在哪，再决定用继承、组合还是模板。技术选择应服务变化维度。
49. **性能认知**：不要把 virtual 当洪水猛兽。先测量热点，再做结构调整。
50. **总结套路**：讲机制（vtable）+ 讲语义（替换原则）+ 讲工程实践（组合优先、最小接口）。
