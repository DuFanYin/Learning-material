# C++ 卡片 06：并发（50）

1. **并发 vs 并行**：并发强调任务交替推进，并行强调同时执行。面试回答先区分概念会更清晰。
2. **线程创建成本**：线程不是“免费协程”，创建和销毁都有系统成本。短任务应考虑线程池。
3. **`std::thread` 生命周期**：线程对象析构前必须 `join` 或 `detach`。否则程序会 `terminate`。
4. **`join` 语义**：等待线程完成并回收资源。常用于明确同步点。
5. **`detach` 风险**：分离线程后失去管理能力。对象生命周期和退出时机更难控制。
6. **共享状态危险**：多个线程读写同一数据会有数据竞争风险。先减少共享，再谈加锁。
7. **数据竞争定义**：至少一个写且无同步的并发访问即 data race。结果是未定义行为。
8. **竞态条件**：程序结果依赖执行时序。不是所有竞态都构成 UB，但都应被设计处理。
9. **互斥锁作用**：一次只允许一个线程进入临界区。保证复合操作原子性。
10. **`std::mutex`**：最基础互斥原语。手动 `lock/unlock` 易漏解锁，推荐 RAII 锁包装。
11. **`lock_guard`**：构造加锁、析构解锁，语义最简单。适合固定作用域互斥。
12. **`unique_lock`**：更灵活，可延迟加锁、手动解锁、与条件变量配合。代价是略高复杂度。
13. **死锁四条件**：互斥、占有且等待、不可剥夺、循环等待。破坏任一条件可避免死锁。
14. **锁顺序策略**：多锁场景保持统一加锁顺序。是最实用的死锁预防手段。
15. **`std::scoped_lock`**：可一次安全锁多个 mutex，内部处理死锁规避。C++17 起很实用。
16. **临界区最小化**：锁内只放必要操作，耗时计算放锁外。可显著降低争用。
17. **条件变量定位**：用于“等待某条件成立”，避免忙等。要和互斥锁配合使用。
18. **`wait` 必须配谓词**：防止虚假唤醒造成逻辑错误。典型写法是 `cv.wait(lk, pred)`。
19. **`notify_one` vs `notify_all`**：前者唤醒一个等待者，后者唤醒全部。按工作模型选择，避免惊群。
20. **生产者消费者模型**：条件变量 + 队列是经典实现。重点是队列条件检查与锁粒度。
21. **原子类型价值**：对简单共享变量可免互斥锁。避免数据竞争并降低部分开销。
22. **`std::atomic` 并非万能**：原子只保证单变量操作原子性。多个变量不变式仍需锁或更高级协议。
23. **内存序基础**：`relaxed/acquire/release/seq_cst` 决定可见性与重排约束。正确性优先于极限性能。
24. **`seq_cst`**：最强顺序语义，最容易推理。默认值通常是安全起点。
25. **`acquire-release`**：通过同步点建立 happens-before。常用于发布-订阅模式。
26. **`relaxed` 场景**：只需原子性不需顺序保证时使用。例如统计计数器。
27. **happens-before**：并发推理核心关系。若无此关系，看到旧值是合法的。
28. **可见性问题**：一个线程写入后，另一个线程未必立刻可见。同步原语负责建立可见性。
29. **指令重排**：编译器和 CPU 都可能重排。并发代码不能依赖“看起来的执行顺序”。
30. **false sharing**：不同线程写同一缓存行不同变量也会互相拖慢。可用填充/对齐分离热点数据。
31. **无锁不等于更快**：无锁算法实现复杂且易错。多数业务场景锁+良好设计更稳。
32. **ABA 问题**：CAS 场景下值从 A->B->A 可能误判“未变化”。需版本号或标记指针解决。
33. **自旋锁适用性**：短临界区低竞争可用。高竞争或长等待会浪费 CPU。
34. **读写锁**：读多写少场景可提升并发度。写饥饿与升级问题需关注。
35. **线程池价值**：复用线程、平滑吞吐、限制并发度。是服务端高频组件。
36. **任务队列背压**：生产速度高于消费速度会堆积。需限流、丢弃策略或优先级调度。
37. **取消与停止**：长任务应支持停止信号。C++20 `jthread` + `stop_token` 提供更好语义。
38. **`std::future` 与 `async`**：适合一次性异步结果获取。复杂任务图常需更专门框架。
39. **异常跨线程传播**：线程内异常不能自动冒泡到主线程。要通过 `promise/future` 或错误通道传递。
40. **日志并发问题**：多线程日志输出易交错。应使用线程安全日志器或批量缓冲。
41. **并发测试困难**：并发 bug 往往与时序相关且不稳定复现。需要压力测试和工具辅助。
42. **ThreadSanitizer**：可检测数据竞争。并发模块开发期应尽早接入。
43. **锁竞争分析**：通过 profile 看等待时间而非只看 CPU。高等待常提示锁粒度过粗。
44. **不可变数据思路**：尽量让共享数据只读，写时复制。可大幅降低同步复杂度。
45. **消息传递模型**：线程间传消息而非共享可变状态。可减少锁并提高可维护性。
46. **并发接口文档**：明确线程安全级别（完全安全/外部同步/线程不安全）。这是工程协作基础。
47. **原子+锁混用**：要清楚边界，避免“看似都加了保护但语义断裂”。统一同步策略更稳。
48. **析构与并发**：对象析构前必须终止其后台线程访问。否则极易出现 UAF。
49. **面试分析套路**：先找共享写，再找同步点，再判断是否存在 data race。最后给修复策略。
50. **并发设计优先级**：先保证正确，再优化吞吐与延迟。错误的高性能没有意义。
