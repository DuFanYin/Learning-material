# C++ 卡片 07：性能与工程（50）

1. **性能三要素**：延迟、吞吐、资源占用要一起看。单点优化常会牺牲其他指标。
2. **先测量后优化**：没有 profile 的优化很可能在“错地方发力”。数据驱动是工程基本功。
3. **热点优先**：优先优化 Top 热点路径。边缘路径再快，对整体提升也有限。
4. **Amdahl 定律**：可优化部分占比决定上限收益。先找最大占比瓶颈。
5. **算法级收益最大**：复杂度从 O(n^2) 到 O(n log n) 往往远超微优化。先从算法层动手。
6. **常数项也重要**：在中小规模数据下，常数与缓存行为会主导性能。不能只看大 O。
7. **缓存局部性**：连续访问比随机访问更易命中 cache。数据布局常比指令级技巧更关键。
8. **`vector` 常胜原因**：连续内存 + 预取友好。很多遍历场景显著优于链表。
9. **分支预测**：高随机分支会导致 pipeline 冲刷。可通过重排条件与数据分桶改善。
10. **函数内联**：减少调用开销并暴露更多优化机会。过度内联会增大代码体积。
11. **代码体积权衡**：I-cache 压力上升会抵消部分收益。性能优化要平衡“更快”与“更大”。
12. **拷贝开销识别**：大对象频繁拷贝是常见隐性成本。接口设计要控制值传递边界。
13. **移动语义收益**：资源转移常比深拷贝便宜。容器重分配路径尤其受益。
14. **`noexcept` move**：标注后容器更敢在扩容时走移动路径。可减少回退到拷贝的概率。
15. **返回值优化**：依赖编译器 RVO 能减少临时对象。别过早手写“看似优化”的 move。
16. **预分配策略**：`reserve`、对象池等可降低分配器压力。高频路径常有明显收益。
17. **内存分配成本**：频繁小分配会造成锁竞争和碎片。批量分配或池化更稳。
18. **对象池边界**：池化适合生命周期短且类型稳定对象。滥用会增加复杂度和内存常驻。
19. **避免不必要抽象层**：高频路径过度虚层会带来分发和定位成本。抽象应与变化需求匹配。
20. **虚函数开销认知**：单次开销通常小，关键是热循环中是否阻碍内联。需结合 profile 判断。
21. **模板静态绑定**：可去掉运行时分发并提升优化空间。代价是编译时和二进制体积上升。
22. **数据驱动设计**：把逻辑组织成连续数据批处理。常比面向对象散布访问更快。
23. **AoS vs SoA**：数组的结构体便于封装，结构体的数组便于 SIMD 和缓存线利用。按访问模式选。
24. **false sharing 规避**：热点计数器放不同缓存行。可用对齐与填充降低伪共享。
25. **锁竞争成本**：高并发下锁等待会吞噬吞吐。要监测等待时间而非只看 CPU。
26. **锁粒度设计**：粒度太粗影响并发，太细增加复杂度。以真实争用数据为准调优。
27. **批处理思维**：把小任务合并可减少调度和同步开销。吞吐系统特别受益。
28. **系统调用成本**：syscall 比普通函数调用贵得多。可通过缓冲与批量 I/O 降低频率。
29. **I/O 与 CPU 解耦**：异步 I/O 或生产消费队列可隐藏等待时间。提升整体流水线效率。
30. **编译优化等级**：`-O2/-O3` 对性能差异显著。性能测试应在接近生产编译参数下进行。
31. **LTO/PGO 价值**：链接时优化和采样引导优化可提升跨模块性能。适合关键产线版本。
32. **调试构建误判**：Debug 结果不代表 Release 性能。面试中要主动指出构建配置影响。
33. **基准测试规范**：固定输入、预热、重复测量、统计分布。只看一次耗时不可靠。
34. **基准污染因素**：CPU 频率调节、后台负载、NUMA 亲和性都会影响结果。要尽量控制环境。
35. **尾延迟意识**：P99 往往比平均值更反映用户体验。服务端优化要关注长尾。
36. **吞吐-延迟权衡**：更高吞吐可能带来排队延迟。目标指标不同，方案也不同。
37. **缓存友好容器选择**：性能关键路径优先连续容器。链式结构仅在必要语义下使用。
38. **字符串处理优化**：避免不必要拷贝与重复分配。`string_view` 可减少临时字符串构造。
39. **`string_view` 生命周期**：它不拥有数据，只是视图。引用失效缓冲区会出悬垂问题。
40. **小对象优化（SOO/SSO）**：把小数据放对象内部可省堆分配。常见于字符串和函数封装对象。
41. **异常与性能**：不抛异常路径通常成本可控。高频失败分支不应依赖异常做普通控制流。
42. **日志开销管理**：同步磁盘/格式化常是隐藏热点。可按级别裁剪并使用异步日志。
43. **性能回归防线**：关键路径建立基准门禁。避免“功能没变但速度慢了”。
44. **可观测性**：指标、trace、profile 结合才能定位瓶颈。单一数据源常导致误判。
45. **可维护性权衡**：极致优化代码难读难改。要确保收益值得复杂度成本。
46. **工程化优化流程**：定位瓶颈 -> 设假设 -> 做实验 -> 验证收益 -> 回归测试。闭环比灵感更重要。
47. **回退策略**：优化上线应可快速回滚。性能改动也可能带来稳定性风险。
48. **跨平台差异**：不同编译器/CPU 结果可能不同。关键优化要做平台验证。
49. **面试表达模板**：先给瓶颈证据，再给方案，再给量化结果。避免“我觉得会快”的主观表述。
50. **终极原则**：正确性和可维护性是前提，性能优化是在前提之上的增量收益。
